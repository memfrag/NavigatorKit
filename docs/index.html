Why a router at all

SwiftUI gives you the pieces, not the trip

Presentations won't batch

Set a tab, a path, and two nested sheets in one mutation and SwiftUI will drop most of it on the floor. Each presentation level needs its own transaction — which is why real apps end up with asyncAfter chains and magic delays.

NavigationPath is a black box

You can push into it, but you can't ask what's inside. "Pop to the product screen if it's already open" or "which window is showing this order?" are unanswerable questions.

Deep links cut across everything

A notification tap may need to switch tabs, rewrite a stack, open a sheet, and pick which window handles it. That logic doesn't belong scattered across views.

The state model

One observable tree per window

Every scene owns a single source of truth: a recursive @Observable tree. SwiftUI containers bind to it — a sheet's content is itself a full navigation context, so "a sheet containing a stack that presents another sheet" is just a deeper tree. Paths are [AnyRoute], not NavigationPath, so the router can read them: pop-to-existing, window matching, and diffing all fall out of inspectability.

SceneNavigator — one per window
TabsLayout — selection: .shop · tabs may host stacks or split views
NavigationContext — path: [.list, .detail(42)]
PresentedContext — .sheet(detents: [.medium]) → content is a NavigationContext again
RoutedAlert — alerts and dialogs attach at any level
The executor

Intents compile to stages

A pure planner diffs your intent against the current tree and buckets it into stages: tear down conflicting presentations, apply tab + path in one transaction, attach one presentation per nesting level, then overlays. Between stages the executor waits for the actual SwiftUI transition — signalled by the routed views, bounded by a timeout — so nested presentations land reliably instead of racing the animation. A newer intent supersedes the in-flight one at its next stage boundary.

Decoupling

Features never meet each other

A feature module owns its route types and registers route → view mappings. It imports NavigatorKit and nothing else — not the app shell, not its sibling features. One registration feeds view resolution, Codable decoding for restoration, and the route's default presentation. Views see exactly one API: the environment navigator.

Deep links are a table

DeepLinks.swift
static let map = DeepLinkMap {  URLPattern("/products/:id/review") { params in    let id = try params.id(Int.self)  // typed, throwing    return NavigationIntent {      SelectTab(AppTab.shop)      SetStack(.list, .detail(id: id))      Present(ReviewRoute.compose(productID: id))    }  }  URLPattern("/settings/**") { _ in    NavigationIntent { SelectTab(AppTab.settings) }  }}

Custom-scheme URLs are normalized so shopapp://products/42 and https://shop.example.com/products/42 share one pattern. Most-specific match wins; a handler that throws (malformed id) falls through to the next pattern. Cold launch is handled — a URL that arrives before any window exists is parked and claimed by the first scene to register.

The contract

Dismissal semantics you can rely on

navigate(to:placement:) resolves placement as: explicit argument → the route type's registered default → .push.

PlacementBehavior
.pushPush onto the deepest active context — the sheet's stack if a sheet is up.
.replaceStackReplace the active context's path with just this route.
.present(style)Present over the active context — sheet with detents, or full-screen cover.
.activateExisting(else:)If the route is anywhere in the scene, reveal it: select its tab, dismiss what covers it, pop to it. Otherwise apply the fallback.
  • Sibling tabs' stacks are never touched by navigation.
  • Presentations are window-global in SwiftUI, so switching tabs dismisses whatever the outgoing tab had presented — deliberately, not accidentally.
  • Mutating a stack that has something presented over it dismisses the presentation first. No invisible changes behind a sheet.
Beyond one window

Windows and relaunches are first-class

SCENES Multi-window routing

Each window instantiates its own state tree from the blueprint. An app-level coordinator routes incoming intents through a pluggable policy — reuse the active window, reuse the window already showing that route (inspectable trees make this a query, not a hack), or always open a new one. Callers can override per call with scenePreference: .newWindow(); single-window platforms fall back gracefully.

RESTORE State restoration

Pass a restorationKey and the whole tree — tab selection, paths, sidebar selection, presentation chains — persists per-window and restores on launch. Routes from renamed or unregistered types degrade gracefully: the path truncates before the first undecodable element and anything stacked above is dropped. Snapshots are versioned from day one.
